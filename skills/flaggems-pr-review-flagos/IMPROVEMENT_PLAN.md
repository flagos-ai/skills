# Code Review Skill 改进方案

## 问题总结

### 1. 算子名显示为 "unknown"
**原因**：
- `fetch_pr_diff.py` 已经检测到算子名并输出在 JSON 的 `operator` 字段
- `review_operator.py` 使用了这个字段进行检查
- 但 `post_review.py` 没有接收这个信息，而是自己从 findings 的 file 路径中重新提取
- 提取逻辑不够健壮，导致显示 "unknown"

**当前代码** (`post_review.py` 第 102-107 行)：
```python
# Try to detect operator from findings context
files = {f.get("file", "") for f in findings}
op = "unknown"
for p in files:
    if "src/flag_gems/ops/" in p and "__init__" not in p:
        op = p.split("/")[-1].replace(".py", "")
        break
```

### 2. 评论格式不够友好
**当前格式**：
```markdown
**[ERROR]** `KERNEL_HEADER`: First line does not start with '# Generated by KernelGen'

**Fix:** Add '# Generated by KernelGen' as the first line if this is auto-generated.
```

**问题**：
- 太机械化，不像人类 reviewer 的语气
- 缺少上下文说明
- 严重级别标签太突兀
- 规则 ID 对用户意义不大

## 改进方案

### 方案 A：最小改动（推荐）

def load_findings(path: str) -> tuple[str | None, list[dict[str, Any]]]:
    """Load findings from JSON file, return (operator, findings)."""
    with open(path) as f:
        data = json.load(f)
    
    # Support both old format (array) and new format (dict with operator)
    if isinstance(data, list):
        return None, data
    elif isinstance(data, dict):
        return data.get("operator"), data.get("findings", [])
    else:
        raise ValueError("Invalid findings JSON format")

#### 2. 优化评论格式

**修改 3**: 让评论更像人类 reviewer

```python
def format_comment_body(findings: list[dict[str, Any]]) -> str:
    """Format multiple findings into a single comment body."""
    parts: list[str] = []
    
    severity_emoji = {
        "error": "🚫",
        "warning": "⚠️",
        "info": "💡"
    }
    
    severity_prefix = {
        "error": "Issue",
        "warning": "Suggestion",
        "info": "Note"
    }
    
    for f in findings:
        severity = f.get("severity", "info")
        rule_id = f.get("rule_id", "UNKNOWN")
        message = f.get("message", "")
        suggestion = f.get("suggestion", "")
        
        emoji = severity_emoji.get(severity, "")
        prefix = severity_prefix.get(severity, "Note")
        
        # 更友好的格式
        part = f"{emoji} **{prefix}**: {message}"
        
        if suggestion:
            part += f"\n\n**Recommended fix:**\n{suggestion}"
        
        # 在底部添加规则引用（小字）
        part += f"\n\n<sub>Rule: `{rule_id}`</sub>"
        
        parts.append(part)
    
    return "\n\n---\n\n".join(parts)
```

**效果对比**：

**改进前**：
```markdown
**[ERROR]** `KERNEL_HEADER`: First line does not start with '# Generated by KernelGen'

**Fix:** Add '# Generated by KernelGen' as the first line if this is auto-generated.
```

**改进后**：
```markdown
🚫 **Issue**: First line does not start with '# Generated by KernelGen'

**Recommended fix:**
Add '# Generated by KernelGen' as the first line if this is auto-generated.

<sub>Rule: `KERNEL_HEADER`</sub>
```

## 推荐实施步骤

1. **第一步**：修改算子名传递逻辑（方案 A 修改 1-2）
   - 改动小，风险低
   - 立即解决 "unknown" 问题

2. **第二步**：优化评论格式（方案 A 修改 3）
   - 让评论更友好
   - 保持规则 ID 但放在不显眼的位置

3. **可选**：如果还不满意，再考虑方案 B 的模板化

## 需要讨论的问题

1. 是否需要保留规则 ID？
   - 保留：方便调试和追溯规则
   - 去掉：更简洁，但失去可追溯性
   - 折中：放在小字或折叠区域

2. emoji 是否合适？
   - 优点：视觉上更友好
   - 缺点：可能显得不够专业

3. 是否需要为每个规则定制消息模板？
   - 优点：更自然
   - 缺点：维护成本高

4. 评论语气：
   - 当前：直接指出问题
   - 可选：更委婉（"Consider...", "It would be better if..."）
